# MacSurf Version History

Per-version milestone narrative for shipped builds (v0.2 baseline through v0.5 fetch recovery). Each entry describes what landed, why it was hard, and the architecture decisions behind the fix. Cross-references commit hashes and `fixesNNN` numbers from [docs/changelog-fixes.md](changelog-fixes.md).

CLAUDE.md keeps only the **current state** and **last shipped fix**; everything else is here.

- **v0.4.5 OFFSCREEN COMPOSITE on G4 / OS 9.2.2 (2026-05-16).** Reached at fixes77g. Animation paint goes through an offscreen GWorld back-buffer; pixels swap to screen via single CopyBits per tick. **Cascade of unrelated bugs collapsed** in one round:
  - **No more white flash on animation ticks.** Old frame stays on screen until the new frame is fully composited in the GWorld, then both swap atomically.
  - **Scroll bar tracking is smooth.** Previously janky / sometimes unresponsive, turns out the underlying culprit was the same plotters.c clip bug (every draw was clipped to empty during certain port states, breaking control redraws too).
  - **URL bar text input works on the initial window.** The 2026-04-18 survey hypothesis ("content redraw overdraws URL rect while TextEdit is still functional internally") was wrong, the actual root cause was that content redraws were rendering to empty clip, so the URL bar's chrome wasn't getting properly preserved and TextEdit's focus dispatch was confused.
  - **URL bar submit navigates pages.** Same root cause class.
  - **Walker-pruning optimisation works.** With `ub=(152,107,432,389)` as the box-tree-walker clip, NetSurf prunes `skips=24` of 58 visited boxes, only the dirty bbox's branches actually fire plotters. Free perf for animation ticks.
  - **The root-cause one-liner**: plotters.c's three `GetPort → (WindowRef)port → GetWRefCon` sites all assumed the current QD port was the window. fixes77f's offscreen GWorld broke that assumption mid-redraw, casting a GWorld pointer to WindowRef returns garbage from WRefCon, `gw->content_rect` reads as `(-32538, 11128, 31871, 0)`, SectRgn produces empty regions, all clips resolve to `(0,0,0,0)`. fixes77g routes those sites through `macos9_find_gw_for_plot()` which reads from `macos9_paint_gw` (set by main.c around `browser_window_redraw`) first. See Known Gotchas entry.
  - **Architecture** (fixes77f, refined by fixes77g):
    1. BeginUpdate sets window port's visRgn to update region.
    2. `GetRegionBounds(visRgn)` → `update_bounds` (window-coord bbox of dirty area).
    3. Reuse / reallocate per-window GWorld matching `content_rect` size.
    4. `SetGWorld(gw, NULL)` + `SetOrigin(content_rect.left, content_rect.top)` so GWorld port coords match window coords.
    5. **No SetClip on the GWorld**, plotters.c manages QD clip exclusively via plot_clip callbacks.
    6. `EraseRect(&update_bounds)` clears only the dirty bbox in the GWorld.
    7. `browser_window_redraw(..., &update_bounds_as_struct_rect, ...)`, passes update_bounds as the walker clip; NetSurf prunes branches whose bbox doesn't intersect, collapsing per-rect animation ticks to ~5 boxes.
    8. `SetGWorld back to window`, then `CopyBits(content_rect → content_rect)`. Window's visRgn is still BeginUpdate's update region, so QD intrinsically clips the blit to dirty pixels.
  - **Conversation that diagnosed it**: SetOrigin coordinate-math sanity-check (does it ADD or ASSIGN?) caught a mental inversion in the V2 proposal before shipping. SetOrigin ASSIGNS, `SetOrigin(0, 38)` means port coord (0, 38) IS pixel (0, 0). Erase-after-SetOrigin with window-coord rect is correct because QD subtracts the origin to find pixel coords.
  - **Speed status**: works perfectly, animation is a little rough framerate-wise on the G4. The walker prune already collapses each tick to ~5 boxes, the remaining cost is QuickDraw's per-pixel rotation math + the per-tick CopyBits. Both have well-known optimisation paths (rotation lookup tables exist already; CopyBits can be RowBytes-aligned). Speed work is **deferred**, correctness/visual quality came first, optimisation can follow whenever it becomes the bottleneck.
- **v0.4.4 CSS GRID V1 on G4 / OS 9.2.2 (2026-05-16).** Reached at fixes75e. Native CSS Grid layout, `display: grid` + `-macsurf-grid: N` syntax for an N-column equal-width grid with row-major auto-placement:
  - **Lite syntax**: `display: grid; -macsurf-grid: N` (or `-macsurf-grid: N M` for N cols × M rows; rows=0 = auto-rows). Pages saying `display: grid; grid-template-columns: repeat(3, 1fr)` can use `-macsurf-grid: 3` to opt in.
  - **Layout** ([browser/netsurf/content/handlers/html/layout_grid.c](browser/netsurf/content/handlers/html/layout_grid.c)): walks children in DOM order, places row-major into N equal-width columns. Row height tracks the tallest child in each row. Reuses `column-gap` storage (fixes148) for inter-cell spacing on both axes.
  - **Storage**: word 14 bit 4 in `style->i.bits`; packed value `(cols<<16)|rows` in `style->i.macsurf_grid`.
  - **Five-fix bug chain converged in this round**: fixes75 added the property and box-tree plumbing → fixes75b added `BOX_GRID` to `layout_block_find_dimensions` dispatch → fixes75c removed the `ns_computed_display` downgrade that was turning `CSS_DISPLAY_GRID` into `CSS_DISPLAY_BLOCK` before box-construct ever saw it → fixes75d added `BOX_GRID` to four flex-only assertions in `layout_block_context` and `layout_absolute` → fixes75e passed `&item->height` (not `&dummy_h`) to `layout_find_dimensions` so `layout_block_context` would see `AUTO` and finalise the real height, without this fix, row height tracked only padding/border, never the tallest item's content.
  - **Test page**: PROBE G1-G7 in `advanced.html` cover 2-col, 3-col, 4-col with partial last row, 6-col tight grid, variable-height row, nested grid (grid-within-grid), and grid + radial-gradient combined. All seven probes render cleanly.
  - **Out of V1 scope (deferred to V2+)**: `grid-template-columns` / `-rows` full grammar (fr, repeat, minmax, auto), explicit placement (`grid-area`, `grid-row`, `grid-column`), `grid-auto-flow: column | dense`, full-fidelity row-gap independent of column-gap, subgrid, per-cell `align-items` / `justify-items`. V1 covers the most common real-world pattern, "N equal columns, auto rows, children in DOM order", which is `display: grid; grid-template-columns: repeat(N, 1fr)`.
- **v0.4.3 RADIAL-GRADIENT on G4 / OS 9.2.2 (2026-05-16).** Reached at fixes74d. `-macsurf-gradient: radial-gradient(...)` parses and renders natively as a 24-ring concentric `PaintOval` stack:
  - **Parser**: accepts `radial-gradient(c1, c2)` plus optional shape/position prefix (`circle`, `ellipse`, `circle at center`, `closest-side`, etc.). Save-try-rewind colour probe distinguishes "prefix present" vs "first token is a colour".
  - **Storage**: shares the linear-gradient int32 slot via a new radial flag in bit 14. c2's red channel drops from R5 to R4 (16 levels, smeared on decode), visually indistinguishable on 8-bit displays.
  - **Plotter**: `PLOT_OP_TYPE_RADIAL_GRADIENT` paints the box with edge colour then stacks 24 progressively smaller `PaintOval` ellipses converging on the centre colour. Ellipse fits box bounds → non-square boxes get an elliptical gradient automatically (CSS's default `ellipse farthest-corner`).
  - **Latent bug fixed in flight**: `html_redraw_box_has_background` didn't know about `-macsurf-gradient` and was returning false for any element without an explicit `background-color`, dropping the gradient on the floor. Latent since fixes47 because every prior gradient probe happened to also set `background:`. fixes74d adds the SET check.
  - **Test page**: PROBE R1-R4 in `advanced.html` cover bare 2-stop, shape/position prefix, aspect ratio (wide ellipse vs tall), and linear-gradient sanity check (proves the R5→R4 bit-pack shuffle didn't regress linear).
- **v0.4.2 CSS TRANSFORM on G4 / OS 9.2.2 (2026-05-16).** Reached at fixes73e. `-macsurf-transform` parses and renders natively:
  - **rotate()**: any angle. Cardinal angles (0/90/180/270) use exact integer rotation; arbitrary angles use a 91-entry Q15 sin/cos lookup table with quadrant symmetry (182 bytes const). Integer math only, no FPU dependency.
  - **translate(x, y)**: signed int8 px range (±127), one-value and two-value forms both supported.
  - **scale(s)** / **scale(sx, sy)** / **scaleX(s)** / **scaleY(s)**: bare CSS numbers parsed via `css__number_from_lwc_string` (not unit-specifier, that incorrectly treated them as percentages). Stored in Q8.8 in a companion `macsurf_transform_b` storage slot.
  - **Composition**: rotate + translate, scale + rotate, all combinations. Scale applies pre-rotation around the box centre.
  - **Storage**: `macsurf_transform` int32 packs rotation Q10.6 (bits 31..16), tx int8 px (15..8), ty int8 px (7..0). `macsurf_transform_b` int32 packs sx_q88 (31..16), sy_q88 (15..0). Identity sentinel for transform_b is 0x01000100.
  - **Paint-don't-relayout semantics**: layout box stays the original size, the painted rectangle paints scaled/rotated around the box centre. Borders remain at the layout slot (intentional, full border transform is V4).
  - **Plotter clip**: transformed paint uses the wide `content_rect` clip rather than the tight per-box clip, so scale > 1 actually escapes the layout slot edges. Non-transform paint paths keep their tight clip.
  - **Test page**: PROBE T1-T7 in `advanced.html` cover cardinal rotation, arbitrary rotation, translate, rotate+translate, uniform scale, scaleX/scaleY/non-uniform scale, and scale+rotate composition. All seven probes render cleanly.
  - **Cascade**: `css__copy_macsurf_transform` and `css__compose_macsurf_transform` propagate BOTH the rotation/translate field AND the scale field, without this, scale dropped to identity after every cascade compose pass (the bug behind fixes73d).

- **v0.5 FETCH SYSTEM RECOVERY (2026-05-17).** Reached at fixes105. Eight rounds (fixes98→105) brought the fetch system from "stuck after about three pages" to indefinite multi-page browsing. The chain in order:
  - **fixes98**, chunked transfer-encoding decoder (`process_chunked_bytes`, four-state CS_SIZE/DATA/CRLF/TRAILER machine) + auto-follow 3xx redirects (parse `Location:`, emit `FETCH_REDIRECT`). Origin servers like nginx that always reply `chunked + keep-alive` now stay keep-alive eligible. Reverted fixes95's `Connection: close` on direct fetches.
  - **fixes99**, pool the endpoint at `FETCH_FINISHED` dispatch time, not at `macos9_http_free` time. NetSurf core calls `ops.free` lazily; slots routinely linger in `MFS_NOTIFIED` across many `http_setup` calls, so the pool was always empty when a new fetch wanted to take from it. Returning the endpoint immediately when we know it's idle lets the very next `mfs_open` against the same host REUSE.
  - **fixes100**, `macos9_gw_invalidate` honors NetSurf's supplied dirty rect (converting layout coords → window coords, clipping to `content_rect`) instead of unconditionally `InvalWindowRect(window, &content_rect)`. Per-keystroke redraws dropped from ~170 DrawText to single-digit.
  - **fixes101**, removed `InvalWindowRect(window, &content_rect)` from `macos9_handle_key_down` after `browser_window_key_press` returns true. NetSurf already invalidated the precise rects during the keypress (textarea bbox + caret); the trailing whole-content InvalRect was undoing fixes100.
  - **fixes102**, fetcher self-frees via `fetch_free(c->parent)` at every terminal callback (FINISHED, ERROR, REDIRECT). NetSurf core does NOT call `ops.free` itself, every reference fetcher (curl, file, about, data, resource, css_fetcher, javascript) self-invokes `fetch_free`. Without this, slots leaked into `MFS_NOTIFIED` forever and NetSurf's per-content / per-bw active-fetch caps tripped after ~3 pages.
  - **fixes103**, `fetch_remove_from_queues(c->parent)` BEFORE `fetch_free(c->parent)`. fixes102's `fetch_free` alone left a dangling pointer in `fetch_ring`; next `RING_GETSIZE` walked freed memory. The MacsBug screenshot's `all_active=4` was literally counting danglers. Both calls required, in that order, RING_REMOVE uses the struct's r_prev/r_next, then free.
  - **fixes104**, aborted-while-queued cleanup. `mfs_poll_one` and `macos9_http_poll` both skipped `MFS_QUEUED` slots BEFORE checking `c->aborted`. When NetSurf aborts a queued sub-resource fetch (every page navigation has 1-2 of these), the abort flag was set but the slot stayed queued forever and leaked in `fetch_ring`. Reordered: abort check runs first, regardless of state.
  - **fixes105**, stub fetchers (`macos9_fetcher_stubs.c`) self-free. The audit-driven discovery: stub fetchers had the *same* leak class as the pre-fixes102 HTTP fetcher. `stub_send_for` emitted FETCH_FINISHED and marked `ctx->done=true` but never called `fetch_remove_from_queues` + `fetch_free`. Every `resource:default.css` and `resource:internal.css` (fetched 2× per `html_create`) leaked. `stub_poll` now does both cleanups for normal-done and aborted contexts, mirroring file.c / about.c / data.c / resource.c.
- **Performance side effects shipped along the way:** per-keystroke redraw on the duck-search-results page dropped from ~172 DrawText calls to single-digit; sub-resource fetches reuse pooled TCP connections (visible as `ep_pool: REUSE key=frogfind.com:80` in the log); slot indices cycle in the low single digits (`slot=0/64` repeatedly) instead of climbing monotonically.
- **v0.4.6.1 PNG TRANSPARENCY on G4 / OS 9.2.2 (2026-05-16).** Reached at fixes79b. Native per-pixel PNG transparency via **lodepng decoder + `CopyMask` 1-bit mask plotter path** (Path A1). `lodepng.c` (7244 lines, single-file C89-clean PNG decoder with bundled DEFLATE, no zlib dependency) decodes PNG straight into a NetSurf bitmap with real RGBA. A 1-bit alpha mask is built alongside the RGBA pixels at decode time (bit set where alpha >= 128, clear otherwise). The plotter takes the `CopyMask` branch when a bitmap has a mask attached; mask = 0 pixels are not transferred, so the destination (typically the parent box's background) shows through. The magenta-keyed `transparent` CopyBits path is gone, never worked reliably on 32-bit pixmaps. Anti-aliased PNG edges become a binary threshold; **Phase A1.5** (queued) upgrades to `CopyDeepMask` + 8-bit mask for hardware-side per-pixel blending if needed.
- **v0.4.6 IMAGES on G4 / OS 9.2.2 (2026-05-16).** Reached at fixes78r. All five image formats, **PNG / GIF / BMP / TIFF / JPEG**, decode and render. PNG now uses lodepng (real alpha, see v0.4.6.1). **GIF gets per-pixel transparency for free** because QT's GIF importer honours the file's palette transparency at decode time. **TIFF still renders opaque**, QT's `QTNewGWorld(k32ARGBPixelFormat)` returns `cDepthErr (-157)` on this Carbon-era QuickTime build, meaning the OSType-aware pixmap allocation path doesn't exist on OS 9 (it was a QuickTime 6 / Mach-O addition that never landed on this platform). The fixes78r fallback degrades alpha-format QT decodes to vanilla `NewGWorld(32)` + opaque render so they're at least visible. Real TIFF per-pixel alpha would need a TIFF decoder in Path A (libtiff or a small TIFF reader). JPEG / BMP have no file-level alpha so opaque is correct.
