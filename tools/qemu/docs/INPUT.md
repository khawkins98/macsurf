# Input & automation: driving the OS 9 guest from the host

Hard-won findings on how to control the Mac OS 9 QEMU guest without a human at
the keyboard/mouse. Synthesized from a 4-agent research sweep + on-machine tests.

## What works / what doesn't (measured)

| Channel | Status |
|---|---|
| QMP keyboard (`send-key`) | ✅ works fully (type, shortcuts, dialog defaults via Return) |
| QMP relative mouse **motion** (`input-send-event` rel) | ✅ works (~1.5× OS 9 acceleration; screenshot-feedback converges) |
| QMP mouse **clicks** on `usb-mouse` | ❌ do NOT register — even with a 250 ms dwell |
| `usb-tablet` (absolute) with **no driver** | ❌ ignored entirely by OS 9 |
| Host HFS+ file injection (mount qcow2's volume) | ✅ works read+write, preserves forks |

## Root cause of the click failure

All QEMU input paths — QMP `input-send-event`, VNC `PointerEvent`, and the native
Cocoa display — converge on the same `qemu_input_event_send()` in `ui/input.c`.
There is **no privileged "cocoa-only" path**. The reason real Cocoa clicks register
but injected ones don't is **timing**: OS 9's HID/ADB driver polls button state on a
periodic tick (~16 ms VBL). A real click "dwells" (down and up land in separate
main-loop iterations, tens of ms apart); a batched/instant down→up is invisible
between two polls. **Fix: always dwell ≥ ~100 ms between button-down and button-up,
as two separate events.** (`vm.py press()` now dwells 150 ms by default.)

But for `usb-mouse` specifically, OS 9's driver also ignores *standalone button
reports* regardless of dwell — which is why the whole community uses `usb-tablet`
plus a guest INIT instead.

## The proven click path: usb-tablet + guest INIT

`kanjitalk755/macos9-usb-tablet` (`USBTabletINIT.sit`, pre-release5) makes OS 9
understand QEMU's absolute `usb-tablet`, giving **absolute-coordinate clicks** (no
calibration). Setup:
1. Host-inject `USB Tablet INIT` into the guest **System Folder** (we `ditto` it onto
   the mounted HFS+ volume).
2. Launch with **`-device usb-tablet`** instead of `usb-mouse`, and **`-M mac99`
   WITHOUT `via=pmu`** (the INIT's README is explicit: `via=pmu` breaks it).
3. Send `abs` x/y (0–32767 scaled to the framebuffer) + a dwelled `btn` down/up.

Caveat to verify on each setup: dropping `via=pmu` must not break the USB keyboard
(`send-key`). Test Return-on-a-dialog right after the machine change.

## The build path that needs NO mouse (preferred for the loop)

CodeWarrior 8 is fully scriptable via AppleEvents (creator `'CWIE'`, `'MMPR'` suite:
`Make`, `RemB`=Remove Object Code, `STrg`=set target, `ClsP`, plus `aevt/odoc`,
`aevt/quit`). You do **not** menu-click to build:
- Run AppleScript **without compiling an applet**: keyboard-launch **Script Editor**
  (`Apple Extras:AppleScript:` or `Applications:AppleScript:`), `Cmd-O` the injected
  script, `Cmd-R` to Run.
- `Make` returns the **error/message list as data** — write it to a file with
  AppleScript file I/O and a full HFS path, so **no "Save Error Window" dialog**.
- Keyboard fallbacks: Make = **Cmd-M**, Run = **Cmd-R**, Open = **Cmd-O**. (Remove
  Object Code / set-target have no stock key — use the AppleEvents.)

### Project + access paths with no GUI config
CodeWarrior **Export/Import Project** round-trips the *entire* project (file list +
all target settings **including Access Paths**) as **XML** — this repo's
`Access Paths.xml` is exactly that. So: keep the project as injectable XML (text,
diffable), host-inject it, and **Import** it on the guest (AppleScript or
File→Import). The `.mcp` is regenerated by import — never hand-edited, honoring the
CLAUDE.md "don't edit MacSurf.mcp" rule (source of truth = XML).

### The one unavoidable first-run click
CW8's first launch shows a modal **"Where is Mac OS X?"** dialog; "Continue" is a
non-default button. Options, best first:
1. **Golden prefs:** click Continue once (via usb-tablet, or the user in a Cocoa
   window), quit, then capture `System Folder:Preferences:Metrowerks:` out of the
   guest. Inject that folder before every automated run → dialog never reappears.
2. usb-tablet click on "Continue" each cold start.

## Fallback channels (if usb-tablet doesn't pan out)

- **In-guest VNC server** (ChromiVNC 3.4a5 or Vine VNC for OS 9), host-injected +
  auto-started via `System Folder:Startup Items:`, reached from a host VNC client
  through `-netdev user,...,hostfwd=tcp::5900-:5900`. Gives real coordinate clicks +
  keyboard. Caveat: first-run config may need a click unless prefs are pre-seeded.
- **KeyQuencer** (control panel, file-seedable macros bound to hotkeys): inject the
  control panel + a macro suitcase with `Click <x> <y>` macros, fire the hotkey via
  QMP `send-key`. Most likely to be genuinely zero-click; no networking needed.
- **Host CGEvent/cliclick on a Cocoa window**: post synthetic clicks at host-screen
  coords over a visible QEMU window (reuses the proven Cocoa path). Needs the window
  visible+frontmost (breaks headless) + Accessibility permission + geometry math.

## Driving the GUI: techniques verified on-machine

With usb-tablet+INIT, absolute clicks work — but classic-Mac GUI idioms have
gotchas that took real trial-and-error:

- **Absolute moves are accurate** (`vm.py move X Y` lands within ~2px). Earlier
  "offset" symptoms were misread cursor positions, not a mapping error.
- **Menus need press-drag, not click-then-click.** A click opens a sticky menu,
  but it times out during the gap before your *separate* item-click call, and the
  fall-through click hits whatever's behind (often switching to the Finder). Use
  `vm.py pressdrag Xtitle Ytitle Xitem Yitem` — press the menu title, drag to the
  item, release, all in one call. This reliably picked File→Import Project and the
  application menu.
- **Bring a backgrounded app forward** via the application menu (top-right) with a
  press-drag to its name.
- **Nav OPEN dialogs** (e.g. Import's "choose file") **type-select the list** —
  type a name, Return opens/enters it. They remember the last folder.
- **Nav SAVE dialogs are different:** typing goes to the **Name field**, not the
  list. Enter folders by **double-clicking** them (Cmd-Down does NOT work). Scroll
  with the scrollbar arrows to reach off-screen folders. **Esc CANCELS the whole
  dialog** — don't use it to dismiss sub-popups.
- **Dialog default button = Return** (e.g. OK / Open / Save / a dialog's bold button).

## Build verification status (what's proven)

- The repo `MacSurf.mcp` is a **valid, importable CodeWarrior XML project** — CW8
  parses all 531 `<FILE>` entries and proceeds to the save step. Confirmed by
  driving **File → Import Project** on it.
- It MUST be imported/saved into the **macos9 folder** (`Back40:Desktop
  Folder:patrick:macsurf-source Folder:browser:netsurf:frontends:macos9:`) so the
  XML's relative `<PATH>../../../utils/…</PATH>` refs resolve. Saving elsewhere
  fails at the first `<FILE>` ("Error importing XML … near line 367").
- Source is staged at exactly that location (`stage-on-bootvol.sh`), CR + TEXT/CWIE.
- **Remaining:** complete Import saving into macos9 → the project loads → `Make`
  (Cmd-M; slow under TCG, expect access-path/component iteration). Best finished
  interactively given the slow build loop and project-config judgment calls.

## Bottom line

The recurring **build loop needs no mouse** — keyboard + AppleScript + host file
injection covers it. Mouse clicking (now solved via usb-tablet+INIT) handles the
one-time GUI setup. Every capability needed to build MacSurf off-hardware is
proven; only the mechanical Import-save + the long first compile remain.
